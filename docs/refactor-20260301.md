# Refactoring SPEC: Cell Index + Unified Cell Mutations

Date: 2026-03-01
Scope: `mcp_server.py`

## Context

The current server keeps only two pieces of global state:

- `loaded_notebook: Notebook | None`
- `loaded_notebook_path: str | None`

Cell mutation APIs are split into two tools:

- `add_code_cell_to_loaded_notebook(code_content, position=None)`
- `add_markdown_cell_to_loaded_notebook(markdown_content, position=None)`

There is no direct tool for editing an existing cell, and no maintained index to quickly resolve notebook cell subsets (code vs markdown) or stable references after mutations.

## Goals

1. Maintain an index over notebook cells (all, code, markdown).
2. Replace split add-cell tools with one unified `add_cell` tool.
3. Add an `edit_cell` tool for in-place cell content updates.

## Non-Goals

- No change to plain-text conversion format in this refactor.
- No notebook execution/runtime features.
- No multi-user/session isolation beyond current single-process global state.

## Current Problems

- **API duplication**: two add methods with nearly identical validation/error handling.
- **No cell addressing contract**: callers rely on implicit position assumptions.
- **No index maintenance**: any code/markdown lookup requires scanning `loaded_notebook.cells` every time.
- **Harder evolution**: edit operations would duplicate more logic unless mutations are centralized.

## Proposed Design

### 1) Add In-Memory Cell Index State

Add a small index structure in `mcp_server.py`, rebuilt from `loaded_notebook.cells` and refreshed after every mutating operation.

Suggested global state:

```python
cell_index: dict[str, list[int]] | None = None
# shape:
# {
#   "all": [0, 1, 2, ...],
#   "code": [0, 3, 5, ...],
#   "markdown": [1, 2, 4, ...],
# }
```

Suggested helpers:

- `_build_cell_index(notebook: Notebook) -> dict[str, list[int]]`
- `_refresh_cell_index() -> None`
- `_require_loaded_notebook() -> str | None` (returns error string if not loaded)

Index refresh triggers:

- `load_notebook`
- `plain_text_to_notebook_file`
- `add_cell` (new)
- `edit_cell` (new)
- any future mutation tools

Notes:

- Index values are notebook positions (0-based).
- Index is ephemeral and recomputed; no persistence required.
- If an unknown cell type appears, include it in `all` only (do not fail).

### 2) Unify Add Operations into `add_cell`

Add one MCP tool:

```python
@mcp.tool()
def add_cell(
    cell_type: str,
    content: str,
    position: int | None = None,
) -> str:
    ...
```

Behavior:

- Accept `cell_type` in `{ "code", "markdown" }` (case-insensitive).
- Route to notebook implementation:
  - code -> `loaded_notebook.add_code_cell(source=content, position=position)`
  - markdown -> `loaded_notebook.add_markdown_cell(source=content, position=position)`
- Refresh index after success.
- Return normalized success/error messages including updated total cell count.

Validation:

- Reject when no notebook is loaded.
- Reject invalid `cell_type` with actionable message.
- Reject invalid `position` values via explicit bounds check where possible.

Backward compatibility:

- Keep existing add tools temporarily as thin wrappers around `add_cell`.
- Mark wrappers as deprecated in docstrings and README.
- Remove wrappers in a later version (see rollout plan).

### 3) Add `edit_cell` Tool

Add one MCP tool:

```python
@mcp.tool()
def edit_cell(
    cell_index_or_position: int,
    content: str,
    cell_type: str | None = None,
) -> str:
    ...
```

Behavior:

- Edit the cell at the given 0-based position.
- If `cell_type` is omitted:
  - preserve existing cell type.
- If `cell_type` is provided:
  - allow only `code` or `markdown`.
  - if different from current type, convert by replacing the target cell with new type/content at same position.

Implementation options (choose based on `Notebook` API availability):

- **Preferred**: direct in-place mutation if `Notebook` exposes a writable cell structure.
- **Fallback**: reconstruct via plain-text round trip or remove+insert strategy to preserve position.

Validation:

- Reject when no notebook is loaded.
- Reject out-of-range index.
- Reject invalid `cell_type` values.

Post-conditions:

- Notebook cell count remains unchanged after edit.
- Index is refreshed.

## Error Handling Contract

Use consistent error prefix and shape:

- `Error: <reason>` for user-correctable failures.
- Success includes operation + resulting total cell count.

Examples:

- `Error: No notebook is currently loaded. Use load_notebook() first.`
- `Error: Invalid cell_type 'raw'. Allowed: code, markdown.`
- `Edited code cell at position 3. Loaded notebook now has 12 cells.`

## API Surface After Refactor

### New

- `add_cell(cell_type, content, position=None)`
- `edit_cell(cell_index_or_position, content, cell_type=None)`

### Deprecated (kept temporarily)

- `add_code_cell_to_loaded_notebook(...)` -> wrapper to `add_cell("code", ...)`
- `add_markdown_cell_to_loaded_notebook(...)` -> wrapper to `add_cell("markdown", ...)`

## Testing Plan

Unit/integration tests should cover:

1. **Index lifecycle**
   - built on load
   - refreshed after add/edit
   - cleared/reset on load failure paths
2. **`add_cell` behavior**
   - add code and markdown
   - insert at middle and append
   - invalid type / invalid position
3. **`edit_cell` behavior**
   - edit content only
   - edit with type conversion
   - out-of-range index handling
4. **Compatibility wrappers**
   - legacy add tools still work and delegate correctly

## Rollout Plan

1. Introduce index helpers and internal refresh hooks.
2. Add `add_cell` and switch old add tools to wrappers.
3. Add `edit_cell` with strict validation.
4. Update README tool list and argument docs.
5. In next minor/major release, remove deprecated wrappers after migration window.

## Open Questions

- Confirm whether the underlying `Notebook` class exposes direct cell mutation APIs; if not, select fallback edit strategy.
- Decide deprecation window length for legacy add tools (e.g., 1 minor release).
